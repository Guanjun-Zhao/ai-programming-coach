# AI 陪练教练：上下文管理方案

## 核心设计：每个**叶子任务**对应一个独立上下文

左侧任务栏中，**叶子节点**（最上的「功能设计」、各**三级**编码小节、末尾「Debug」）各自对应后台一个独立的对话存储。用户在该叶子下的所有对话都保存在该存储中，**叶子之间**上下文完全物理隔离。

**二级标题（父节点）**不对应任何 AI 对话文件：仅有**复选框完成态**与**折叠展开态**的持久化（见项目方案总结中的 `_tree` 约定）。

---

## 上下文文件结构

每个**叶子**任务的历史只存对话轮次，**不含 system prompt**。System prompt 在每次调用 API 时从 `sections.json` 动态构建，拼在列表开头后传入，不持久化：

```json
// history/task_1_2_1.json  —  只存这些
[
  {"role": "assistant", "content": "好的，接下来我们来进行 Warrior 类的声明。……"},
  {"role": "user",      "content": "前向声明是什么意思？"},
  {"role": "assistant", "content": "……"}
]
```

```python
# 每次调用 API 时在内存中拼接，不写回文件
messages = [system_prompt] + load_history("task_1_2_1")
```

每次用户发送消息，将新的 user/assistant 消息对追加到历史文件；system prompt 始终从 `sections.json` 实时读取，修改 prompt 无需动历史文件。

## 任务切换行为

| 操作 | 行为 |
|------|------|
| 点击左侧「功能设计」且该任务尚无历史 | 初始化该叶子槽；程序**自动**追加一条助手导读（可带一条对用户不可见的占位 user 以满足 API），持久化 |
| 点击左侧某一**编码叶子**且未开始 | 初始化或加载该叶子 `coach_history`；可按产品约定自动追加开场 assistant |
| 点击左侧**二级父节点** | **不**加载、不切换任何 `coach_history`；不改变右侧对话区当前上下文（仅折叠或勾选） |
| 点击左侧进行中的叶子任务 | 加载已有上下文，聊天区恢复至上次对话状态 |
| 点击左侧已完成的叶子任务 | 加载已有上下文，允许继续提问 |

## 验证阶段的代码块注入

验证时需要将参考 `code` 块传给 AI，但**不持久化到上下文文件**。

**不适用验证的节**：`skip_code_verify: true` 或 `role: planning`（「功能设计」导读）——不得将参考 `code` 注入请求（且通常无 `code` 字段）。

具体处理方式：
1. 用户粘贴代码，前端识别为验证请求；
2. 若当前节允许验证：后端临时构建一次 API 调用：在当前上下文列表末尾动态拼入包含参考代码的 system/user 消息；
3. 获得 AI 响应后，**只将用户消息和 AI 回复存入上下文文件**，参考代码块丢弃不保存。

这样可以防止参考代码通过对话历史泄露给用户。

---

## Debug 任务的上下文特殊处理

Debug **叶子**同样拥有独立的上下文文件，但其 System Prompt 与编码节不同，初始化时注入的是调试教练角色指令及全部样例数据（来自 `samples.json`），而非章节 `description`。

**错误分析时的完整代码注入**：当用户输出与期望不一致时，后端临时将中列代码保存区的**完整程序代码**拼入当次 API 请求，用于 AI 分析逻辑错误。与验证阶段的参考代码注入相同，完整代码**不持久化到 Debug 上下文文件**，分析完成后丢弃。

持久化到 Debug 上下文文件的内容仅为：AI 发出的样例输入消息、用户回传的运行结果、AI 的比对反馈与分析引导。
