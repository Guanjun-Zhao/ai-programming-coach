# AI编程教练 · 项目方案总结

> 课程：程序设计实习 · 元学习赛道  
> 赛道定位：辅助程设学习的实用工具——针对魔兽世界大作业的AI引导式代码书写与测试工具  
> 最后更新：2026年5月13日

---

## 一、项目定位

### 一句话描述
一个针对魔兽世界OJ大作业的AI辅助学习工具，提供**逐节引导式代码书写**和**测试样例库**两项核心功能。

### 核心价值
魔兽世界这道题有两个核心痛点：一是**题目逻辑极度复杂**，规则多且相互嵌套，学生难以独立从零完成完整实现；二是**OJ提供的样例数量不足以覆盖所有边界条件**，导致代码通过已有样例后仍可能在未知边界条件上出错。本程序的两项核心功能分别对应这两个痛点：AI教练模式逐节引导学生用自然语言理解并书写代码，样例库模式提供系统性的边界覆盖测试。

### 定位说明
- AI教练的角色是**逐节引导代码书写**，每节用自然语言描述实现逻辑，不直接输出代码
- 参考代码仅在验证阶段临时注入，不持久化，不存在答案泄露风险
- 面向正在完成魔兽世界大作业的同学

## 二、功能设计

### 2.1 整体结构

```
主页
├── 顶部 API 配置栏（模型 + API Key）
└── 四个版本入口（魔兽世界一、二、三、最终版）
    └── 版本页（三列布局）
        ├── 左列：任务进度栏
        ├── 中列：代码保存区
        └── 右列：AI对话区（含教练模式 / 样例库模式切换）
```

### 2.2 主页
- **顶部 API 配置栏**（位于版本入口之上）：下拉选择调用模型，密码框填写 DeepSeek API Key，点击「保存」写入本机 `data/app_settings.json` 并立即作用于全局 AI 调用。默认模型为 **`deepseek-v4-flash`**（DeepSeek V4 Flash）；下拉另提供 `deepseek-v4-pro` 与兼容项 `deepseek-chat`。若本地未保存 Key，运行时仍回退环境变量 `DEEPSEEK_API_KEY` / `DEEPSEEK_MODEL`；均未配置时 AI 对话返回占位回复。
- 四个按钮，对应魔兽世界的四个发布版本
- 每个按钮显示该版本的完成进度：**已勾选复选框数 / 左侧所有复选框总数**（含「功能设计」、各二级标题行、各三级小节、末尾 **Debug**；二级与三级勾选各自独立，见 §2.3）

### 2.3 版本页布局（三列）

**左列（约 1/5 宽）：任务进度栏（树形分组 + 折叠）**

教练文档中，**与 AI 对话一一对应的是三级标题**（`### 1.2.1`、`### 1.2.2` …）；多个三级标题归在同一个**二级标题**（`## 1.2　创建 Warrior 类`）之下。左侧任务栏用 **树形控件**（如 `QTreeWidget`）呈现，而不是扁平列表。

- **最上一条**：固定为 **「功能设计」**，对应章首导入 + 该章「规划」小节（版本一为 `## 1.1　规划程序` 等）。左侧有复选框；**点击该行**切换右侧 AI 对话（导读上下文）。本节无参考代码、不走「粘贴代码验证」流程；首次进入且该任务对话为空时，由程序自动请求 AI 输出导读并以自然口语过渡到下一节（见 §2.4）。**从主页点击版本按钮进入版本页时，默认选中本行**，右侧 AI 对话区自动加载「功能设计」上下文（若历史为空则同样触发自动导读）。
- **中间若干组**：每个 **二级标题** 对应树中一个**父节点**：
  - 父节点左侧有**复选框**，表示「本大块已通读/完成」，持久化；**点击父节点标题区域不切换**右侧 AI 对话（父节点**不是**任务按钮），**不**为该节点单独建立 `coach_history`。
  - 父节点旁提供 **一键折叠/展开**：第一次点击展开其下**全部**三级子行，再点一次全部收起；仅影响展示，不改变子任务对话内容与勾选状态；折叠态写入 `user_data.json` 中该版本下的 `_tree.h2_expanded`（或等价字段），退出程序后保留。
- **子节点（三级标题）**：左侧有复选框；**点击子节点**与现有一致：右侧切换到该小节的独立 AI 对话；勾选表示该编码小节已完成。
- 若某 `##` 节下没有 `###`（如整节 `## 1.5`），则该组下挂**一个**与二级标题同名的子叶子，仍对应独立 `task_id` 与对话。
- **最末一条**：固定为 **「Debug」** 叶子任务；触发条件见 §2.5（「所有编码节」不含「功能设计」与 Debug 本身）。

**中列（约 2/5 宽）：代码保存区**
- 保存**完整程序**的统一代码区，不按节拆分
- 用户随着各节推进，逐步将新完成的代码追加/更新到此区域，最终形成与本地程序完全一致的完整代码
- Debug 阶段出现错误时，AI 会读取此处的完整代码进行逻辑分析

**右列（约 2/5 宽）：AI对话区**
- 根据左列**选中的叶子任务**（功能设计、各三级小节、Debug）切换对应上下文；**点击二级父节点不改变**当前对话上下文。每个叶子任务的对话记录独立保存。

### 2.4 AI教练模式

#### 核心设计思路

开发阶段将参考文档按小节预先切割；**运行时只读**的 `data/sections.json` 按版本组织，推荐**嵌套形状**：顶层为 **「功能设计」导读对象**（`role: planning`，无 `code` 或 `code` 为空）+ **`groups` 数组**（每项为二级标题 `h2_id` / `h2_title` 及其下的 `sections` 叶子列表）。每个叶子与 `sections.json` 中的一条记录对应，例如：

```json
{
  "section_id": "1.2.2",
  "task_id": "task_1_2_2",
  "title": "构造函数",
  "description": "该节的自然语言说明（直接来自参考文档的解释部分）",
  "code": "该节的参考代码块"
}
```

可选字段：`"skip_code_verify": true`（导读节强制为 true）、`"role": "planning"`。`description` 和 `code` 在不同阶段分别使用，互不干扰。AI 在教学阶段看不到 `code`，无法泄露。

#### 导读节（「功能设计」）

- **触发**：用户点击左侧最上「功能设计」叶子任务；若该任务 `coach_history` 为空，程序**自动**发起一轮 API（或等价逻辑），不要求用户先输入。
- **喂给 AI 的内容**：仅该节的 `description`（章首导入 + 规划小节全文），无 `code`。
- **AI 的任务**：用中文复述/整理题意与程序分工，**不要**要求用户在本节提交代码或进入验证流程；**结尾必须**有一句自然过渡（例如引导用户点击下一节开始具体编码）。
- **验证阶段**：**不适用**；用户若粘贴代码，教练可温和提示到下一节再进行代码相关讨论。

#### 两个工作阶段（适用于各「编码」叶子小节）

**阶段一：教学阶段**

触发时机：用户点击左侧某一**三级叶子**任务，或在对话框中提问。

喂给 AI 的内容：仅该节的 `description`，不包含 `code`（`skip_code_verify` 为 true 的节仍只注入 `description`，且不进入阶段二）。

AI 的任务：用自然语言描述这一节需要实现的逻辑，回答用户问题，引导用户动手写代码。

System Prompt 关键指令：
> 你是一名 C++ 编程教练。你手中有当前小节的逻辑说明，但没有参考代码。你的教学方式是用中文自然语言描述"这段代码需要做什么事"、"这里的逻辑是如何流动的"、"这个变量存的是什么信息"。不要输出任何代码块。如果用户要求你直接给出代码，回答"这部分需要你自己动手写，我可以继续描述细节帮你理解"。

**阶段二：验证阶段**

触发时机：用户在对话框中粘贴自己写好的代码（**且**当前节未标记 `skip_code_verify`）。

喂给 AI 的内容：该节的 `code`（参考代码块）+ 用户提交的代码，临时拼入请求，不持久化。

若 `code` 为空或 `skip_code_verify` 为 true：不走本阶段逻辑，由教学提示引导用户进入下一节。

AI 的任务：判断用户代码与参考代码的**逻辑是否等价**（不要求字面一致，变量名、风格不计）。

System Prompt 关键指令：
> 以下是本节的参考代码（内部资料，不得输出给用户）和用户提交的代码。请判断用户代码是否实现了与参考代码等价的逻辑功能。只判断逻辑正确性，不评价代码风格。
> - 如果通过：输出"你的代码已完成这部分的功能。输入「下一步」，让我们继续下一节的书写。"
> - 如果未通过：用自然语言指出缺少或有误的逻辑点，不要给出正确代码。

#### 完成流程（用户侧，全部手动）

验证通过后，用户手动完成以下操作：
1. 将本节代码追加/更新到中列的**统一代码保存区**（中列始终保存完整程序，不按节拆分）
2. 勾选左侧任务栏对应**叶子**节的复选框，标记为完成（二级标题的勾选单独表示大块完成度，与叶子勾选独立）

AI 不参与这两步，避免自动化带来的复杂性。

#### 上下文管理

每个**叶子任务**（功能设计、各三级编码节、Debug）对应后台一个独立的上下文存储，为标准消息列表格式（**二级标题父节点无独立对话文件**，仅持久化勾选与折叠状态）：

```json
[
  {"role": "system",    "content": "（含该节 description 的系统提示）"},
  {"role": "assistant", "content": "好的，接下来我们来进行 Warrior 类声明的书写……"},
  {"role": "user",      "content": "前向声明是什么意思？"},
  {"role": "assistant", "content": "……"}
]
```

每次用户发消息，将新消息追加到文件；每次调用 API，将整个列表作为 `messages` 参数传入。

**验证阶段的代码块注入**不持久化：后端临时构建 API 调用时将参考代码动态拼入，获得响应后只将用户消息和 AI 回复存入上下文文件，参考代码块丢弃，防止通过历史记录泄露。

**任务切换行为：**

| 操作 | 行为 |
|------|------|
| 从主页点击版本按钮进入版本页 | 默认选中「功能设计」叶子任务，自动触发导读逻辑 |
| 点击左侧「功能设计」且该任务尚无历史 | 加载/初始化该任务槽；**自动**写入一条助手导读（含过渡语），持久化 |
| 点击左侧某一**三级叶子**且未开始 | 初始化或加载该任务 `coach_history`；若有「开场引导」约定，可自动追加助手首句 |
| 点击左侧**二级父节点** | **不**切换右侧对话上下文；可折叠/展开子项；可勾选大块完成态 |
| 点击左侧进行中的叶子任务 | 加载已有上下文文件，聊天区恢复至上次对话状态 |
| 点击左侧已完成的叶子任务 | 加载已有上下文文件，允许继续提问 |

### 2.5 Debug 模式

**触发方式**：用户完成**所有编码叶子小节**（即除「功能设计」导读与「Debug」本身以外的各三级编码任务）后，点击左侧任务栏末尾的「Debug」叶子项。

**前提条件**：中列代码保存区已保存完整程序代码。

#### 交互流程

1. AI 从后台预置的 `samples.json` 中取出第一条样例，将样例**输入**发送给用户，要求其在本地运行并回传输出结果。
2. 用户本地运行后，将实际输出粘贴回对话框。
3. **程序**（本地代码）将用户粘贴的输出与 `samples.json` 中该条样例的正确输出做**字符串精确比对**（去首尾空白后对比），无需调用 API：
   - **一致**：程序直接回显「样例 N 通过」，取下一条样例输入发送给用户，重复流程。
   - **不一致**：触发一次 DeepSeek API 调用，进入错误分析流程。
4. 所有样例全部通过后，AI 回复完成提示，用户手动勾选左侧「Debug」复选框。

#### 错误分析流程（输出不一致时）

AI 临时读取中列代码保存区的**完整程序代码**（不持久化），结合当前样例的输入、期望输出、用户实际输出，用自然语言分析逻辑错误位置，引导用户修改。用户修改后重新运行，将新输出回传，AI 重新比对。AI 在此过程中**不直接给出修改后的代码**。

#### 样例数据准备（开发阶段）

`samples.json` 在开发阶段由团队预置，程序运行时只读：

- **老师样例**：手动整理，用AI辅助添加说明标签，标注测试的功能点或边界条件
- **AI生成补充样例**：针对老师样例未覆盖的边界条件，用AI生成后经人工核查写入

每条样例包含：输入、正确输出、说明标签、来源标注。

## 三、技术选型

| 项目 | 选择 | 理由 |
|------|------|------|
| 开发语言 | Python | 赵冠钧最熟，API调用简单 |
| GUI框架 | PyQt | 课程支持，文档丰富，赵冠钧可用Cursor快速上手 |
| AI API | DeepSeek（默认 `deepseek-v4-flash`；主页可配置 Key 与模型） | 成本低，中文支持好，接口与OpenAI兼容 |
| 持久化 | 本地JSON文件 | 够用，实现简单 |
| 版本管理 | GitHub | 课程要求，支持并行开发 |

### 持久化数据结构

数据按**更新频率和职责**拆分为三类文件，互不耦合：

```
data/
  app_settings.json   ← 本机 API Key 与模型选择（已 gitignore，不提交仓库）
  version1/
    sections.json     ← 开发阶段预置，运行时只读，只含版本一内容
    samples.json      ← 开发阶段预置，运行时只读，只含版本一内容
    state.json        ← 只存进度状态，轻量，频繁读写
    code.cpp          ← 完整程序，纯文本，直接可编辑
    history/
      task_1_2_1.json ← 各任务对话历史，独立文件
      task_1_2_2.json
      ...
      debug.json
  version2/
    sections.json
    samples.json
    state.json
    code.cpp
    history/
      ...
  version3/  （结构同上）
  version4/  （结构同上）
```

**`app_settings.json`**（应用级，仅存本机；字段 `api_key`、`model`，默认 `model` 为 `deepseek-v4-flash`）

**`state.json`**（只存完成状态和 debug 进度，不存历史；亦可与 `user_data.json` 合并，由实现选定）

以下为与**树形任务栏**对齐的 **`user_data.json` 中某版本**示意（与现有骨架兼容时可二合一）：

```json
{
  "version1": {
    "_tree": {
      "h2_expanded": { "1.2": true, "1.3": true },
      "h2_completed": { "1.2": false, "1.3": false }
    },
    "task_1_1": { "completed": false, "coach_history": [] },
    "task_1_2_1": { "completed": true, "coach_history": [] },
    "task_debug": {
      "completed": false,
      "coach_history": []
    }
  }
}
```

说明：**`_tree`** 仅存二级标题的折叠与勾选；各 **`task_*`** 叶子存 `completed` 与 `coach_history`。若团队仍拆分 `state.json` / `history/*.json`，语义与本表一致即可。

**`code.cpp`**（中列完整程序，纯文本直接读写，无需 JSON 转义）

**`history/task_1_2_1.json`**（只存对话轮次，**不含 system prompt**）
```json
[
  {"role": "assistant", "content": "好的，接下来我们来进行 Warrior 类的声明……"},
  {"role": "user",      "content": "前向声明是什么意思？"},
  {"role": "assistant", "content": "……"}
]
```

**`history/debug.json`**（只存样例对话轮次，不存样例内容本身）
```json
[
  {"role": "assistant", "content": "所有编码节已完成，现在开始调试。这是第 1 条样例输入：……"},
  {"role": "user",      "content": "……（用户回传的运行结果）"},
  {"role": "assistant", "content": "样例 1 通过，下面是第 2 条样例：……"}
]
```

**`data/version1/sections.json`**（开发阶段预置，运行时只读；每个版本独立一份，**嵌套分组示例**）

```json
{
  "planning": {
    "section_id": "1.1",
    "task_id": "task_1_1",
    "title": "功能设计",
    "description": "（章首导入 + ## 1.1 规划程序 全文）",
    "code": "",
    "skip_code_verify": true,
    "role": "planning"
  },
  "groups": [
    {
      "h2_id": "1.2",
      "h2_title": "创建 Warrior 类",
      "sections": [
        {
          "section_id": "1.2.1",
          "task_id": "task_1_2_1",
          "title": "类的声明",
          "description": "……",
          "code": "……"
        }
      ]
    }
  ],
  "debug_task_id": "task_debug"
}
```

亦允许**扁平 + `h2_id`/`h2_title` 外键**的等价结构，由构建脚本生成；UI 层负责按二级聚组与排序。

**`data/version1/samples.json`**（开发阶段预置，运行时只读；每个版本独立一份）
```json
[
  {
    "input":  "…",
    "output": "…",
    "tags":   ["边界条件：X", "功能点：Y"],
    "source": "老师样例"
  }
]
```

### System Prompt 策略

- System prompt **不持久化**，每次调用 API 时从当前版本对应的 `data/versionN/sections.json` 动态构建后拼入历史记录开头，prompt 需要调整时只改该版本的 `sections.json` 一处即可
- 教学阶段：只注入当前叶子节的 `description`，AI 看不到 `code`（导读节同样只注入 `description`）
- 验证阶段：仅对未标记 `skip_code_verify` 的叶子节临时注入 `code` 用于比对，响应后丢弃
- Debug 阶段：样例比对由本地代码执行（无需 API）；当前样例从 `data/versionN/samples.json` 按 `current_sample_index` 实时读取，不存入历史；仅在比对不一致触发错误分析时才调用 API，临时注入 `code.cpp` 完整代码，响应后丢弃

## 四、项目分工

### 总原则
