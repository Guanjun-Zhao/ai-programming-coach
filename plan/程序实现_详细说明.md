# AI 编程教练 · 程序实现详细说明

> **与代码对齐**：下列描述以仓库当前实现为准（若与早期产品文档不一致，以代码为准）。主要对应文件：`main_window.py`、`version_page.py`、`chat_widget.py`、`ai_coach.py`、`data_manager.py`、`sections_loader.py`。

---

## 1. 项目目标与两条能力线

### 1.1 一句话定位

针对魔兽世界 OJ 大作业的 **AI 引导式代码书写** 与 **测试样例库（Debug）** 的本地学习工具。

### 1.2 两条能力线与痛点对应

| 痛点 | 能力线 | 说明 |
|------|--------|------|
| 题目逻辑极度复杂，规则嵌套多，难以从零独立完成 | **AI 教练模式** | 按小节（与文档三级标题对齐）用自然语言引导理解与书写；**不**在教学中向模型暴露参考代码，降低「代写」与答案泄露 |
| OJ 公开样例不足以覆盖边界，易在隐藏数据上失败 | **样例库 / Debug 模式** | 预置 `samples.json`，逐条让用户本地运行并回传输出，结合完整程序做错误分析 |

### 1.3 角色与安全边界（产品约束）

- AI 教练在教学中**只输出自然语言**，不输出代码块；参考代码仅在**验证阶段**或 **Debug 错误分析** 时临时注入请求，**不写入持久化历史**。
- 参考代码仅在验证阶段临时注入，验证结束后丢弃，避免通过历史文件泄露。

## 2. 界面与信息架构（对照 `main_window.py` / `version_page.py` / `chat_widget.py`）

### 2.1 整体结构（页面层级）

```
主页
├── 顶部 API 配置栏（模型 + API Key +「保存」+ 显示/隐藏密钥按钮）
└── 四个版本入口（文案见 main_window.VERSION_ENTRIES：一·备战 … 四·终极版）
    └── 版本页（VersionPage；三列拉伸比约 1:2:2）
        ├── 左列：任务树（QTreeWidget：分组 + 复选框 + 折叠）
        ├── 中列：完整程序（QPlainTextEdit → data/{version_id}/code.cpp）
        └── 右列：AI 教练对话（ChatWidget；无独立「样例库」Tab）
```

### 2.2 主页

#### 2.2.1 顶部 API 配置栏

- **实现文件**：`main_window.py`。
- **位置**：版本按钮之上。
- **模型**：`QComboBox`，项数据为 `deepseek-v4-flash` / `deepseek-v4-pro` / `deepseek-chat`；磁盘默认见 `data_manager.DEFAULT_APP_MODEL`。
- **密钥**：`QLineEdit` 密码模式；旁有 `QToolButton`「👁」切换明文（`_on_api_key_visibility_toggled`）。
- **保存**：写入 `data/app_settings.json`，并 `ai_coach.set_runtime_config`。若填写了非空 Key，保存后会弹出**模态**进度框，后台线程执行 `ai_coach.ping_api()`（单次超时约 `API_PING_TIMEOUT_SECONDS`，默认 45s）。
- **进入版本页条件**（`_open_version`）：必须 `ai_coach.has_api_key()` 且 `MainWindow._api_verified_ok`（最近一次 ping 成功）。否则 `QMessageBox` 提示，**不会**进入 `VersionPage`。
- **启动**：若磁盘已有 Key，`QTimer.singleShot(0, _startup_api_verify_if_needed)` 自动 ping；失败则版本按钮禁用。
- **Key/模型优先级**（`ai_coach`）：`set_runtime_config` 非空值优先；否则 Key 读 `DEEPSEEK_API_KEY`；模型读 `DEEPSEEK_MODEL`，再否则 `deepseek-v4-flash`。
- **占位回复**：无 Key 时 `chat` 返回 `_missing_key_reply`；因校验未通过无法进版本页，故「在版本页无 Key 仅占位」不是主路径；`ChatWidget` 会在曾写入占位导读且后来补上 Key 时重试（`refresh_bootstrap` + `_is_planning_placeholder_bootstrap` 等）。

#### 2.2.2 版本入口与进度

- 按钮文案：`_refresh_home_progress_labels` 设为 `{VERSION_ENTRIES 中文标题}  [{num}/{den}]`。
- **分子/分母**：`sections_loader.progress_numerator` / `progress_denominator` + `data_manager.load_version_state`（见 §9 `state.json`）。

### 2.3 版本页三列

| 列 | 布局 | 实现 |
|----|------|------|
| 左 | `addLayout(..., 1)` | `QTreeWidget` `_task_tree` |
| 中 | `addLayout(..., 2)` | `QPlainTextEdit`，`textChanged` → `save_version_code` |
| 右 | `addLayout(..., 2)` | `ChatWidget`，`set_task(tid)` |

### 2.3.1 样例数据与「样例库」

- 当前 UI **无**「教练 / 样例库」双 Tab；样例仅在选中 **Debug 叶子** 且 `_on_send_debug` 时使用。
- 路径：`data/{version_id}/samples.json`（`load_version_samples`）。根目录 `data/samples.json` 与 `load_samples()` **当前未被对话/UI 调用**（可为脚本预留）。

### 2.4 左列任务栏

#### 2.4.1 与文档结构的对应关系

- `sections.json` 的 `planning`、`groups[].sections[]`、`debug_task_id` 驱动树；与 Markdown 层级的对应由内容维护。
- 控件为 **`QTreeWidget`**（`version_page.py`）。

#### 2.4.2 最上叶子：「功能设计」

- 有独立 `task_id`、复选框；默认 `setCurrentItem` 到 planning（若存在）。
- **自动导读**（`chat_widget._bootstrap_planning_if_needed`）：磁盘历史为空时，有 Key 则后台线程发触发句 **`请开始本节导读讲解。`**（写入 `user` + `assistant`）；无 Key 则同步 `chat` 并可能得到 `[占位回复]`。

#### 2.4.3 二级父节点 + 三级子叶子

- **父节点**：`ROLE_TASK` 为空、`ROLE_H2` 为 `h2_id`；勾选写入 `state.json` → `_tree.h2_completed`；展开写入 `_tree.h2_expanded`。
- **点击父节点**：`currentItemChanged` 若 `ROLE_TASK` 非非空字符串则 **return**，**不**调用 `ChatWidget.set_task`，右侧保持上一叶子。
- **子节点**：`set_task` 切换 `history/{task_id}.json`。
- **编码小节空历史**：触发句为 **`请开始本节的学习引导讲解。`**（`_bootstrap_coding_if_needed`）。

#### 2.4.4 Debug 叶子

- `debug_task_id` 决定 `task_id`；树项标题显示为「Debug」。
- **未硬性禁用**：可随时点开；若编码叶子未全完成，首条 assistant 仅追加**建议先勾选**的提示（`all_coding_leaves_completed`），不阻止调试。

#### 2.4.5 其它

- 返回主页：`VersionPage.back_requested` → 栈索引 0；刷新主页进度。
- 再次进入同一版本：`_open_version` 末尾调用 `refresh_bootstrap()`，便于主页保存 API 后补发导读。

## 3. 数据模型：`sections.json` 与叶子记录

### 3.1 运行时只读与按版本隔离

- **加载路径**（`sections_loader.get_version_spec`）：若存在 `data/{version_id}/sections.json` 则只读该文件；否则从根文件 `data/sections.json` 中取键 `version_id`（与主页 `VERSION_ENTRIES` 的 id 一致）。带 mtime 缓存。
- **开发阶段**预置，程序运行时 **只读**；`get_leaf_section` 等读取 `description` / `code` / `role` / `skip_code_verify`。

### 3.2 推荐嵌套形状（权威示例）

顶层包含：

1. **「功能设计」导读对象**（`role: planning`，无 `code` 或 `code` 为空）；
2. **`groups` 数组**：每项为二级标题的 `h2_id`、`h2_title` 及其下的 `sections` 叶子列表。

单条 **叶子** 与 AI 对话一一对应，字段示例：

```json
{
  "section_id": "1.2.2",
  "task_id": "task_1_2_2",
  "title": "构造函数",
  "description": "该节的自然语言说明（直接来自参考文档的解释部分）",
  "code": "该节的参考代码块"
}
```

**常用可选字段**

| 字段 | 含义 |
|------|------|
| `skip_code_verify` | `true` 时强制不走验证阶段（导读节为 `true`） |
| `role` | 如 `"planning"` 表示导读节 |

**字段使用边界**

- `description` 与 `code` **在不同阶段分别使用**，互不干扰。
- **教学阶段**：模型 **只能** 看到 `description`，**不能** 看到 `code`（除非实现错误）。
- **验证阶段**：临时把 `code` 与用户粘贴代码一并送入模型做逻辑等价判断。

### 3.3 完整文件形状示例（嵌套 + planning）

```json
{
  "planning": {
    "section_id": "1.1",
    "task_id": "task_1_1",
    "title": "功能设计",
    "description": "（章首导入 + ## 1.1 规划程序 全文）",
    "code": "",
    "skip_code_verify": true,
    "role": "planning"
  },
  "groups": [
    {
      "h2_id": "1.2",
      "h2_title": "创建 Warrior 类",
      "sections": [
        {
          "section_id": "1.2.1",
          "task_id": "task_1_2_1",
          "title": "类的声明",
          "description": "……",
          "code": "……"
        }
      ]
    }
  ],
  "debug_task_id": "task_debug"
}
```

### 3.4 等价扁平结构

亦允许 **扁平列表 + `h2_id` / `h2_title` 外键** 的等价结构，由构建脚本生成；**UI 层**负责按二级聚组与排序。实现时二选一或内部统一转换为内存树即可。

## 4. AI 教练：导读节（「功能设计」）

### 4.1 触发条件

1. 用户点击左侧最上「功能设计」叶子；**或** 从主页进入版本页默认落在此叶子。
2. 若 `data/{version}/history/{task_id}.json` 为空列表，则触发自动导读（见下）。

**实现注意（`chat_widget.py`）**

- 自动导读写入 **可见的** `user` 触发句：导读为 `请开始本节导读讲解。`；编码小节为 `请开始本节的学习引导讲解。`，再写助手回复。
- 有 Key 时用 `IntroBootstrapThread` 异步请求；无 Key 时同步 `ai_coach.chat`，可能得到 `[占位回复]`。

### 4.2 请求内容

- **仅** 该节的 `description`（章首导入 + 规划小节全文）。
- **无** `code` 注入。

### 4.3 模型行为（产品要求）

- 用中文复述/整理题意与程序分工。
- **不要** 要求用户在本节提交代码或进入验证流程。
- **结尾必须** 有一句自然过渡（例如引导用户点击下一节开始具体编码）。
- 若用户粘贴代码：教练可温和提示到下一节再讨论代码与验证。

### 4.4 验证阶段

- **不适用**；`skip_code_verify: true` / `role: planning`。

### 4.5 System Prompt 组成（导读，与 `ai_coach.build_task_system` 一致）

1. **基底**：`prompts/{version_id}.txt`（`get_system_prompt`；缺失则占位说明）。
2. **追加**：`【当前任务：{title}（导读/功能设计）】` + `description` + 固定教学要求（只讲题意与分工、不要代码块、不要本节验证、结尾过渡）。

（早期《提示词设计方案》中的长段 quote 与上述语义一致，可对照。）

## 5. AI 教练：编码叶子的两阶段

### 5.1 阶段一：教学阶段

**触发**

- 用户点击左侧某一 **三级叶子**；或用户在对话框中提问（当前叶子已选定）。

**请求内容**

- **仅** `description`。
- **不包含** `code`。
- 若节标记 `skip_code_verify: true`，仍只注入 `description`，且 **不进入** 阶段二。

**模型任务**

- 用自然语言说明本节要实现什么、控制流、变量含义等。
- **禁止** 输出代码块。
- 用户索要完整代码时，引导其自己动手写，可继续讲细节。

**System Prompt 组成（教学）**

基底 `prompts/{version_id}.txt` + `【当前编码小节：{title}（{section_id}）】` + `description` + 固定教学要求（描述逻辑、不要代码块、索要代码时引导自己动手）。

### 5.2 阶段二：验证阶段

**触发（同时满足，`ai_coach.should_verify`）**

1. `_looks_like_code_snippet`：strip 后长度 ≥ 40、含换行，且含 `;` / `{` / `#include` 之一。
2. 当前节未标记 `skip_code_verify`，且 `role` 不为 `planning` / `debug`。
3. 当前节 `code` strip 后非空。

**若不满足**

- `code` 为空或 `skip_code_verify: true`：不走本阶段，由教学话术引导进入下一节。

**请求内容（`chat_verify`）**

- `build_verify_system` 把参考 `code` 与用户整段消息放进 **system**；HTTP **user** 为固定短句：`请验证我提交的代码是否与本节要求逻辑等价。`
- **不附带多轮历史**：`chat_verify` 使用 `_complete(system, user)`，不读取 `history/*.json`。
- 参考 `code` 不出现在历史文件中。

**模型任务**

- 逻辑等价判定；不得泄露参考代码原文（与产品文档一致）。

**验证 system 要点**（`build_verify_system`）：角色说明 + 本节标题 + 参考/用户代码块 + 通过/未通过固定句式（实现里为中文引号 `「」`）。

### 5.3 验证阶段实现步骤（与 `ai_coach.chat` / `ChatWidget` 一致）

1. `should_verify` 为真时，`chat` 直接 `return chat_verify(...)`，**不拼接**历史消息参与 API。
2. `chat_verify`：`build_verify_system` + 单次 `_complete`。
3. `ChatWidget._on_send`：先持久化用户消息，再 `CoachChatThread` 调 `chat`；回调校验末尾 user 一致后追加 assistant。
4. 参考代码仅存在于当次请求的 system，**不写入** `history`。

## 6. 用户完成流程（全部手动）

验证通过后（或用户自认完成教学阶段后），由用户 **手动** 完成：

1. 将本节相关代码 **追加/合并** 到中列 **统一代码保存区**（始终表示 **完整程序**，不按节拆分文件粒度）。
2. 勾选左侧对应 **叶子** 复选框。二级父节点勾选 **单独** 表示大块完成度，与叶子勾选独立。

**设计原因**：不由 AI 或程序自动勾选/自动合并代码，避免状态机复杂与误操作。

## 7. 上下文管理（专章）

### 7.1 核心原则

- **每个叶子任务**（功能设计、各三级编码节、Debug）对应 **一个** 独立对话存储；叶子之间 **物理隔离**。
- **二级父节点** 没有独立 `coach_history` 文件；仅有 **复选框完成态** 与 **折叠展开态**（`_tree`）。

### 7.2 历史文件内容：`history/*.json`

- 只存 **对话轮次**（`user` / `assistant` 等），**不含** `system`。
- **教学**：`system` = `build_task_system`（`prompts/{version_id}.txt` + 本节 `description`）；`messages` = `[{"role":"system","content":system}] + load_task_history(...) + 本轮 user`（`_complete_with_history`）。
- **验证**：见 §5.3，无历史。
- **Debug 错误分析**：`build_debug_analysis_system`（基底 + 样例与输出 + 完整程序）。

```python
# 教学路径语义
payload = [{"role": "system", "content": build_task_system(...)}] + load_task_history(...)
```

- `history/*.json` 仅 `user`/`assistant`；每次请求时现组 `system`。

### 7.3 任务切换行为表（合并去重）

| 操作 | 行为 |
|------|------|
| 从主页点击版本按钮进入版本页 | 默认选中「功能设计」；右侧加载该叶子上下文；历史为空则触发自动导读 |
| 点击「功能设计」且尚无历史 | 初始化该叶子槽；**自动** 写入一条助手导读（可配合占位 `user`），持久化 |
| 点击某一 **三级编码叶子** 且未开始 | 初始化或加载该任务 `coach_history`；可按约定自动追加助手开场白 |
| 点击 **二级父节点** | **不** 切换右侧对话；不加载任何 `coach_history`；可折叠/展开；可勾选大块完成态 |
| 点击进行中的叶子 | 加载已有历史，恢复聊天区 |
| 点击已完成的叶子 | 加载已有历史，**允许继续提问** |
| 主页保存 API 后再次进入版本 | `refresh_bootstrap()` 可重试占位导读 |

### 7.4 Debug 任务的上下文（与 §8 配合）

- Debug 历史文件：`history_filename(task_id)` —— 仅当 `task_id == "task_debug"` 时为 `debug.json`，否则为 `{task_id}.json`（应与 `debug_task_id` 配置一致）。
- **首条 assistant**：`_bootstrap_debug_if_needed` **本地生成**（不调 API），含首条样例输入；`current_sample_index` 存于 `state.json` 中该 debug 任务 slot。
- **错误分析**：临时将中列 **完整程序** 拼入当次 API；**不** 持久化到 Debug 历史。
- **持久化内容** 仅为：AI 发出的样例输入说明、用户回传的运行结果、AI 的通过/失败反馈与排查引导（不含完整 `code.cpp` 快照、不含 `samples.json` 全量镜像）。

## 8. Debug 模式（以《项目方案总结》为权威）

### 8.1 与《提示词设计方案》的差异说明（必读）

| 文档 | 表述 |
|------|------|
| 《提示词设计方案》 | 由 AI 将用户输出与期望输出做字符串比对 |
| 《项目方案总结》 | **本地程序** 比对；一致时 **不调用** API；不一致时才调用 API 做错误分析 |

**本实现说明采用结论**：**字符串比对由本地代码完成**；API 仅用于 **错误分析**（及必要时生成下一条样例提示话术——若下一条样例输入也由本地模板拼接，则甚至可不调用 API 推进下一条）。助手侧「样例 N 通过」可由程序生成固定文案 + 本地读取下一条输入展示。

### 8.2 触发与前提（与 `chat_widget` 一致）

- **触发**：用户点击 Debug 叶子即可（无硬性禁用，见 §2.4.4）。
- **前提（软）**：提示用户用中列完整程序在本地运行。

### 8.3 主循环（推荐实现逻辑）

1. 从 `data/{version_id}/samples.json` 取当前样例（`state.json` 中 debug 任务的 `current_sample_index`）。
2. 由程序（或一次 API）向用户展示 **第 N 条样例的输入**，要求用户在本地编译运行并将 **标准输出** 粘贴回对话框。
3. **程序** 使用 `data_manager.normalize_program_output`：统一 `\n`、整体 strip、**每行行尾空白 strip** 后比较。
4. **一致**：程序直接回显「样例 N 通过」，索引 +1，回到步骤 2；**此步不调用 DeepSeek**。
5. **不一致**：调用一次 DeepSeek（或当前配置模型），进入 **错误分析**（§8.4）。
6. 全部样例通过后，由程序或模型输出完成提示；用户 **手动** 勾选 Debug 复选框。

### 8.4 错误分析流程

- 临时读取中列 **完整程序**，与当前样例的 `input`、`output`、用户实际 `output` 一并作为上下文发给模型。
- 模型用自然语言分析可能逻辑错误位置，**不直接给出修改后的完整代码**。
- 分析完成后：**不** 把完整程序写入 `debug.json`。

### 8.5 Debug 相关 System Prompt（调用 API 时使用）

当需要模型参与（例如错误分析、或团队选择由模型朗读下一条样例）时，可使用《提示词设计方案》中的 Debug 指令作为 **行为约束**；其中要求模型「逐字符比对」的句子应视为 **已被本地比对替代**，实现时建议改为：

- 明确告知模型：**比对结果已由程序得出为不一致**；
- 附上 **期望输出** 与 **用户输出** 及 **样例输入**；
- 附上 **完整用户代码**（临时）。

《提示词设计方案》原文（保留供参考，实现时需按 §8.1 裁剪）：

> 你是一名 C++ 调试教练。你将逐条给用户提供测试样例输入，用户在本地运行后将结果回传给你。你手中有每条样例的正确输出，请逐字符比对用户的实际输出与正确输出。
> - 如果一致：回复"样例 N 通过，下面是第 N+1 条样例："并给出下一条样例输入。
> - 如果不一致：结合用户完整程序代码（见下方）、样例输入、期望输出和实际输出，用自然语言分析哪段逻辑可能导致了这个差异，引导用户排查和修改。不要直接给出修改后的代码。

### 8.6 `samples.json` 形态与准备流程

**运行时只读**，开发阶段预置：

```json
[
  {
    "input":  "…",
    "output": "…",
    "tags":   ["边界条件：X", "功能点：Y"],
    "source": "老师样例"
  }
]
```

- **老师样例**：人工整理，可用 AI 辅助写 `tags`。
- **AI 生成补充样例**：覆盖老师未覆盖的边界，**人工核查** 后写入。

### 8.7 `history/debug.json` 示例（只存对话轮次）

```json
[
  {"role": "assistant", "content": "所有编码节已完成，现在开始调试。这是第 1 条样例输入：……"},
  {"role": "user",      "content": "……（用户回传的运行结果）"},
  {"role": "assistant", "content": "样例 1 通过，下面是第 2 条样例：……"}
]
```

说明：若「样例 N 通过」完全由本地生成，文件中仍可存一条 `assistant` 以保持 UI 历史连贯，语义与项目方案一致即可。

      task_1_2_1.json

## 10. System Prompt 总策略（与代码一致）

- **不持久化** 到 `history/*.json`。
- **基底**：`prompts/{version_id}.txt`。
- **教学**：`build_task_system` 追加 `description`；模型看不到 `code`。
- **验证**：`build_verify_system`；`code` 仅在当次 system。
- **Debug**：首条引导本地生成；比对在 `ChatWidget._on_send_debug`；不一致时 `build_debug_analysis_system` 临时带完整程序。

## 11. 产品 / 实现决策速查表

| 问题 | 决策 |
|------|------|
| 教学时是否给 AI 看参考代码？ | 否，只给 `description` |
| 验证时用 `code` 还是 `description` 做基准？ | 用 `code` 块比对逻辑等价 |
| 泄露风险如何控制？ | System 标注内部资料；参考代码与完整代码均不写入历史 |
| 任务勾选和代码更新是否自动化？ | 否，用户手动 |
| AI 完成信号如何设计？ | 固定自然语言提示，无需机器可读标记 |
| 中列代码保存方式？ | 统一完整程序 |
| Debug 样例比对方式？ | **本地**字符串精确比对（可配置空白/换行归一）；不一致再 API |
| Debug 是否直接给修改代码？ | 否 |
| 二级标题是否参与 AI 对话？ | 否；仅复选框 + 折叠 |
| 「功能设计」是否走代码验证？ | 否 |
| 左侧任务栏结构？ | 树：导读叶子 + 按 `##` 分组的 `###` 叶子 + Debug |
| 验证触发？ | `_looks_like_code_snippet` 启发式 |
| 验证是否带历史？ | **否**（`chat_verify` 单轮） |
| 清空对话？ | 「清空聊天记录」按钮；Debug 会重置 `current_sample_index` |

## 12. 实现层风险与缓解（与程序行为相关）

| 风险 | 应对（实现向） |
|------|----------------|
| System Prompt 验证误判 | 第四阶段迭代；初版允许偶发误判 |
| Debug 用户输出格式不规范 | 比对逻辑集中在本地模块，便于迭代；UI 提示用户完整复制输出 |
| AI 生成样例质量 | 全部人工审核后再入库 |
