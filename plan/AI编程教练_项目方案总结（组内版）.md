## 一、项目定位

### 一句话描述
一个针对魔兽世界OJ大作业的AI辅助学习工具，提供版本化任务追踪、代码逻辑对照分析和测试样例库（含AI补充生成）三项核心功能。

### 核心价值
魔兽世界这道题有两个核心痛点：一是**题目逻辑极度复杂**，规则多且相互嵌套，学生难以逐一厘清自己的实现是否符合要求；二是**OJ提供的样例数量不足以覆盖所有边界条件**，导致代码通过已有样例后仍可能在未知边界条件上出错。本程序的两项核心功能分别对应这两个痛点：AI教练模式帮助学生理解逻辑要求，样例库模式提供系统性的边界覆盖测试。

### 定位说明
- AI教练的角色是**辅助理解和测试**，不是判断代码对错
- 程序不包含参考代码，不存在答案泄露风险
- 面向正在完成魔兽世界大作业的同学

## 二、功能设计

### 2.1 整体结构（四层）

```
主页
└── 四个版本入口（魔兽世界一、二、三、最终版）
    └── 版本页
        ├── 左侧：任务清单（可勾选，勾选状态持久化）
        └── 右侧：交互区
            ├── AI教练模式（代码逻辑对照分析）
            └── 样例库模式（样例展示）
```

### 2.2 主页
- 四个按钮，对应魔兽世界的四个发布版本
- 每个按钮显示该版本的完成进度（已勾选任务数/总任务数）

### 2.3 版本页布局
- **左侧（1/5宽）**：任务清单
  - 每条任务前有复选框，完成后可打勾
  - 每条任务文本同时是按钮，点击后右侧切换到该任务的交互区
  - 勾选状态退出程序后保留
- **右侧（4/5宽）**：交互区
  - 顶部有两个切换按钮：「AI教练」和「样例库」
  - 每个任务在两种模式下各自有独立的聊天记录，持久化保存

### 2.4 AI教练模式
**功能定义**：代码逻辑对照分析，不做对错判断

**交互流程**：
1. 用户在对话框中输入内容（与豆包、DeepSeek等对话产品交互方式相同）
2. AI从三个维度给出回应：
   - 解释用户这段代码实现了什么逻辑
   - 列出题目对该部分的完整要求（来自System Prompt中的结构化规则描述）
   - 基于前两点推断出可能存在的遗漏或不一致之处（例如"你的代码在X情况下可能未处理Y"），供用户自行核查

**明确不做的事**：
- 不作最终裁决（不说"你的代码是正确的/错误的"）
- 不直接给出修改后的代码
- 不包含参考代码

**界面**：标准聊天界面（对话记录 + 底部输入框）

### 2.5 样例库模式
**数据来源**：
- **老师样例**（标注"来源：老师"）：开发阶段由团队手动整理，用AI辅助分类，生成带标签的JSON文件，程序直接读取展示
- **AI生成样例**（标注"来源：AI生成"）：开发阶段由团队针对老师样例未覆盖的边界条件使用AI生成，经人工核查后写入`samples.json`，程序运行时同样直接读取展示，不在运行时动态生成

**浏览方式**：点击左侧任务后，右侧显示该任务对应的所有样例，三列卡片布局平铺展示，不做筛选。

**展示格式**：每个样例以卡片形式展示，包含：
- 样例输入
- 样例输出
- 说明标签（这条样例测试的是哪个功能点或边界条件）
- 来源标注

**样例标注流程**（开发阶段完成，非运行时）：
1. 团队通读所有老师样例
2. 用AI（Claude/Gemini）为每条样例生成说明标签，注明其测试的功能点或边界条件
3. 人工核查标签准确性，整理为`samples.json`
4. 标记哪些边界条件尚未被老师样例覆盖，由团队用AI补充生成对应样例并核查后写入

## 三、技术选型

| 项目 | 选择 | 理由 |
|---|---|---|
| 开发语言 | Python | 赵冠钧最熟，API调用简单 |
| GUI框架 | PyQt | 课程支持，文档丰富，赵冠钧可用Cursor快速上手 |
| AI API | DeepSeek | 成本低，中文支持好，接口与OpenAI兼容 |
| 持久化 | 本地JSON文件 | 够用，实现简单，大一同学5分钟能上手 |
| 版本管理 | GitHub | 课程要求，支持并行开发 |

### 持久化数据结构（示意）

程序使用两个独立的JSON文件，职责分离：

**`user_data.json`**（存用户运行时状态，读写）
```json
{
  "version1": {
    "task1": {
      "completed": true,
      "coach_history": [
        {"role": "user", "content": "..."},
        {"role": "assistant", "content": "..."}
      ]
    }
  }
}
```

**`samples.json`**（开发阶段预置，程序运行时只读）
```json
{
  "version1": {
    "task1": [
      {
        "input": "...",
        "output": "...",
        "tags": ["边界条件：X", "功能点：Y"],
        "source": "老师样例"
      },
      {
        "input": "...",
        "output": "...",
        "tags": ["边界条件：Z"],
        "source": "AI生成"
      }
    ]
  }
}
```

### System Prompt策略
- 参考代码**仅用于开发阶段**，帮助团队理解正确逻辑
- 开发者将正确逻辑转化为结构化自然语言，写入System Prompt
- 程序中**不包含参考代码**，无学术诚信风险
- 每个版本的AI教练有独立的System Prompt，包含该版本对应的题目规则描述
- **不要求全员啃透四题细节**：至少由负责 Prompt 的同学对照 `problems/*.md` 与跨版本易混点维护规则正文；其余成员可按分工做浅层对照（见下「重要前提」）

### 重要前提（题目理解如何分工）

- **目标**：保证写入产品的 **System Prompt、样例标签、任务文案** 建立在可验收的规则之上；**不追求**组员人人都会手写魔兽世界 AC。
- **真理链（专人）**：赵冠钧负责规则正文与 Prompt 草稿，对照仓库 `problems/*.md` 与 [魔兽世界_跨版本易混点.md](魔兽世界_跨版本易混点.md)；版一至版三可与 `answer/` 参考实现对照逻辑走向（仅开发阶段，交付物仍不含参考代码）。
- **浅层对照（协作者）**：李佳昱做样例时与赵冠钧对齐跨版本易混点；师圣晴写任务清单时对照题目中的功能划分，不确定处由赵冠钧答疑或指向原题段落。
- **全员最低线**：三人对本方案中的分工、接口与时间节奏对齐；**是否各自通读四版全文**由组内自定，但**各内容产出的责任人**须能在交付前对照原题或真理链完成核对。
